# Simintero Platform — Phased Production Roadmap

**Date:** 2026-06-19
**Basis:** `auditfull06192026.md` (code-level audit @ `d5ad2fa`)
**North star:** a **real (not mock) design-partner pilot** of the integrated prior-authorization value chain — *Digicore decides what's covered and why → Revital reads the documents and assembles grounded evidence → Enstellar orchestrates the governed human lifecycle → Qualitron measures quality.*
**Sequencing under this north star:** `0 → 1 → 2 → 3` is the critical path; the **pilot is demonstrable at the end of Phase 3**. Phases 4–5 (quality breadth, conformance, security/deploy hardening) follow. (If the driver ever shifts to a *compliance/deployable* posture, pull the Phase 5 edge-auth/secrets/deploy items forward to right after Phase 0 — that is the one real fork.)

---

## Framing: this is ONE platform, ONE value chain

Simintero is the core foundation — substrate, domain/canonical models, commons, eventing, multi-tenancy — and the four products are not independent apps; they **compose** into a single customer-facing capability. The roadmap therefore deepens the *integrated chain*, not four parallel products:

```
        ┌─────────── Simintero substrate (RLS, event plane, VKAS, terminology, canonical model) ───────────┐
        │                                                                                                    │
  PA request → [Digicore] is it covered? what evidence? ──► [Revital] read docs, extract & ground evidence ──►
              [Enstellar] orchestrate completeness → review → determination → appeals ──► [Qualitron] measure quality
```

## Regulatory scope — two CMS rules (0057-F + 0053-F)

The platform now targets **two** CMS interoperability mandates that share machinery but are **distinct workflows** — do not conflate them:

- **CMS-0057-F — Prior Authorization (the pilot spine, Phases 0–3).** Coverage decision + the PA lifecycle. Standards: FHIR PAS (`Claim/$submit`/`$inquire`), CRD/DTR, X12 278/275, US Core/Da Vinci. **PA electronic attachments stay here** (see carve-out).
- **CMS-0053-F — Attachments Rule (claims documentation, new Phase 4).** Structured electronic exchange of clinical documentation an insurer requests to **evaluate and process a healthcare *claim*** (post-service / payment integrity) — medical records, imaging, labs, clinical notes, telehealth docs. **This is the CLAIMS attachment workflow, not prior auth.** Key elements:
  - **Technical framework:** X12 **Version 6020** transactions **275** (attachment response) + **277** (Request for Additional Information — the solicited-attachment trigger), carrying **HL7 C-CDA** documents per the **2022 Attachments Implementation Guide**; LOINC attachment-type codes.
  - **Electronic signatures:** a mandatory e-signature standard tied directly to the attachment data, authenticating sender identity.
  - **PA carve-out:** CMS removed prior authorization from 0053-F after industry pushback — **PA attachments remain under 0057-F** (handled in Phase 3 via PAS/275). So 0053-F is scoped to **claims only**.

**How 0053-F fits without re-platforming:** it reuses the substrate (Phase 0), the hardened document/attachment layer, and Revital extraction incl. C-CDA (Phase 2) — adding only X12 275/277 v6020 maps, C-CDA + Attachments-IG validation, electronic signatures, the solicited 277→275 request/response correlation, and a claims-adjudication-side documentation trigger. It does **not** need Digicore coverage decisioning (Phase 1) or the PA appeals lifecycle (Phase 3), so it **branches off Phase 2** and is sequenced **after the PA pilot** under the design-partner north star (it can parallelize with Phase 3 if a separate team/compliance deadline warrants).

## Sequencing principles

1. **Fix the shared floor before adding floors.** The dead substrate event plane, unproven tenant isolation, and absence of integration tests in CI are inherited by *all four* products. Phase 0 is non-negotiable and first.
2. **Value chain over breadth.** Digicore + Revital + Enstellar working together *is* the product. Make that chain real before widening to Qualitron breadth and the full CMS API set.
3. **Vertical slices, not horizontal.** Take 1–2 real coverage scenarios (e.g. knee-arthroscopy 27447 + one imaging/lab criterion) *all the way* through real CQL-vs-FHIR → real document extraction → completeness → determination → appeal, rather than half-building every capability.
4. **Every phase leaves the system both more real and more production-trustworthy**, and is independently demonstrable.

## Dependency spine

`0 → 1 → 2 → 3` is a hard chain (Revital needs real Digicore evidence rules; Enstellar completeness gating needs both real). **Phase 4 (CMS-0053-F claims attachments)** branches off **Phase 2** (document + C-CDA + Revital extraction) — independent of Digicore decisioning and the PA lifecycle, so it can run after the pilot or in parallel with Phase 3. **Phase 5 (Quality)** depends on Phase 1 (shared CQL engine). **Phase 6 (hardening)** can largely parallelize once Phase 0 is done.

**Why first:** you cannot trust any product increment built on a dead event plane, unproven tenant isolation, and mock-only tests.

## Phase 1 — Real decisioning core (Digicore depth)

**Goal:** move from boolean-criteria gates to **real CQL-against-FHIR-data** — the single largest functional gap in the platform.
**Why now:** every downstream product value (Revital evidence targets, Enstellar completeness, Qualitron measures) depends on Digicore expressing real clinical criteria.

- **Real ELM evaluation** against a FHIR data model: `Retrieve`, value-set membership (via terminology-service `$expand`), intervals/quantities/temporal logic. *(closes §3 "boolean-criteria not CQL-vs-FHIR")*
- **Terminology depth** — real `$expand` engine + value-set packs (VSAC sync or seeded) + `ConceptMap`/`$translate`, so membership actually works. *(closes §3 DIG-TRM shallowness)*
- **Versioning completeness** — rollback + real content diff in VKAS; fix the `rolled_back`/`superseded` enum mismatch; persist the rules trace (drop `newTraceRef()`). *(closes §3 DIG-VER/DIG-TRC gaps)*

**Exit criteria:** a coverage rule authored as CQL that retrieves Conditions/Observations from case FHIR data and tests value-set membership evaluates correctly, deterministically, end-to-end, with a persisted trace; rollback works.
**Demo at end of phase:** "a real clinical criterion (not a pre-distilled boolean) evaluates against case FHIR data with a value-set check and an auditable trace."

## Phase 2 — Real evidence (Revital depth)

**Goal:** replace mock AI with real grounded extraction so Revital delivers actual value on real packets, feeding Digicore's now-real evidence requirements.
**Why now:** Digicore (Phase 1) now declares real evidence targets; Revital can ground against them.

- **Real ingestion** — OCR + classification + segmentation with page/span provenance; **parse structured HL7 C-CDA documents** (sections/entries) as a first-class input alongside PDF/scan — the document format CMS-0053-F mandates, reused by both PA and claims attachments. *(closes §4 REV-ING stub; enables Phase 4)*
- **Real extraction** — entity extraction + terminology normalization (calling terminology-service) + FHIR resource output with `Provenance`; resolve `trc_pending` to the gateway audit event. *(closes §4 REV-EXT fake normalization, REV-TRC placeholder)*
- **Real grounded summarization** against a boundary-authorized model behind the model-access layer — keep the already-good enforced citation + abstention rails.
- **Evidence-to-criteria mapping** driven by Digicore's real evidence requirements → structured completeness/**conflict** results Enstellar consumes. *(closes §4 REV-EVD `conflicts:[]`)*
- **Eval/model-ops gate** — gold-standard eval + drift monitoring + canary/rollback + clinical sign-off (at minimum the promotion gate). *(closes §4 REV-EVAL/MOPS placeholders)*

**Exit criteria:** a real scanned PA packet → grounded cited summary + coded extracted evidence + completeness result, evaluated against a real Digicore rule, passing an eval gate, with zero ungrounded assertions.
**Demo at end of phase:** "upload a real faxed packet; Revital returns a cited summary and coded evidence that maps to Digicore's requirements, with gaps and conflicts flagged."

## Phase 3 — Complete the UM lifecycle (Enstellar depth)  ◄ PILOT MILESTONE

**Goal:** the governed human-in-the-loop lifecycle is complete and compliant — closing the value chain end to end. **This is the design-partner pilot.**

- **DTR/CQL completeness gating** before determination (now real because Digicore + Revital are real). *(closes §2 ENS-DOC)*
- **Pend/RFI** driven by real completeness gaps; **notifications fire on ALL determinations** incl. adverse/human-decided (fix the `DECISION_RECORDED` gap — a regulatory-notice risk). *(closes §2 ENS-COM gap)*
- **Appeals & grievances** (ENS-APP — the Not-started v1 must): appeal intake, regulatory clocks, independent reviewer + COI, overturn/uphold, reopening.
- **Live SLA breach monitoring** + escalation. *(closes §2 ENS-CLK)*
- **Metadata-driven clocks + workflow stages** per tenant/LOB — replace hardcoded constants. *(closes §2 ENS-CFG/ENS-WF)*
- **AI model/prompt version capture** on the case. *(closes §2 ENS-AI provenance gap)*

**Exit criteria:** a full PA lifecycle — intake → completeness/RFI → review (real AI + real rules trace) → determination (adverse with sign-off + notice) → appeal → closure — runs **configurably for two LOB profiles** with clocks honored.
**Demo at end of phase (the pilot):** "a design partner runs real PA cases end to end: real coverage decisioning, AI-assisted grounded review, human determination with compliant notices, and an appeal — configurable, auditable, no code forks."

## Phase 4 — Claims attachments (CMS-0053-F)

**Goal:** support the CMS-0053-F Attachments Rule — structured electronic exchange of clinical documentation to evaluate and process **claims** (post-service / payment integrity), reusing the PA machinery.
**Why here:** a distinct, regulator-mandated workflow that branches off Phase 2 (real document + Revital extraction + C-CDA) and the substrate; sequenced after the PA pilot but ahead of quality breadth because it carries a compliance deadline. (Can parallelize with Phase 3 if resourced separately.)

- **X12 v6020 275/277 translation** — extend the interop/canonical mapping to **X12 Version 6020** transactions **277** (Request for Additional Information — the solicited-attachment trigger) and **275** (attachment response). Lossless bidirectional mapping through the canonical model; raw-payload retention for replay. *(extends ENS-INT / Appendix A.7; PA today uses 278/275 — 6020 is a new version line)*
- **Solicited request/response correlation** — the 277-RFAI request (which documents, identified by LOINC attachment codes) ↔ the 275 response, correlated to the originating claim; support unsolicited attachments too.
- **HL7 C-CDA + 2022 Attachments IG** — ingest/validate C-CDA documents per the HL7 Attachments Implementation Guide; hand C-CDA to Revital for grounded extraction (reusing Phase 2); retain provenance.
- **Electronic signatures** — implement the mandated e-signature standard tied to the attachment data, authenticating sender identity end to end; verify on receipt and persist signature provenance to the audit trail. (Reusable trust capability for partner endpoints.)
- **Claims-side trigger & integration** — wire the documentation-request trigger from the claims-adjudication path (the `claims` module / core-admin integration) so a claim pending documentation emits a 277-RFAI; route the returned 275/C-CDA evidence back to the adjudication / payment-integrity reviewer. Enstellar handles the exchange + document layer; it does **not** adjudicate the claim. *(gives the currently-unexercised `modules/claims` a real purpose — §6)*

**Exit criteria:** a claim pending documentation triggers a 277-RFAI; the provider returns an e-signed 275 carrying a C-CDA; the platform validates the signature + C-CDA, extracts grounded evidence via Revital, and surfaces it to the reviewer — correlated to the claim, fully audited, configurable per tenant/LOB.
**Demo at end of phase:** "a claim requests additional documentation electronically; an e-signed C-CDA attachment returns via X12 6020 275, is validated and extracted, and is attached to the claim — no fax."

## Phase 5 — Quality & standards conformance (Qualitron real + Inferno)

**Goal:** real digital measurement on the now-real fabric, and provable standards conformance.

- **Real measure engine** — execute Digicore-governed CQL `Measure`/`Library` (reusing the Phase 1 engine) → real FHIR `MeasureReport` (num/denom/exclusion/exception). **Make QUAL-LOGIC true** (authored in Digicore, executed in Qualitron). *(closes §5 bespoke-SQL-checker + false QUAL-LOGIC)*
- **Deploy reporting + aggregation** modules; evidence viewer + provenance; supplemental-data ingestion + re-evaluation loop; DEQM gaps-in-care + one program's submission-ready output; gap→Enstellar closure + reconcile. *(closes §5 unmounted modules)*
- **Conformance in CI** — stand up Inferno (US Core + Da Vinci CRD/DTR/PAS) as a **gating** suite; fix the broken conformance workflow; CapabilityStatement accuracy. **Add CMS-0053-F conformance:** X12 6020 275/277 round-trip, C-CDA + 2022 Attachments IG validation, and electronic-signature verification. *(closes §6 conformance present-but-not-run)*

**Exit criteria:** a measure runs from Digicore-governed CQL over the shared fabric producing a real `MeasureReport`; Inferno PAS/CRD/DTR suites pass in CI.
**Demo at end of phase:** "a HEDIS-style measure runs from a Digicore-governed CQL Measure over the shared fabric; the platform passes Inferno PAS/CRD/DTR."

## Phase 6 — Production hardening & GA

**Goal:** security, deploy, scale, resilience, deployment-tier, ops maturity.

- **Edge security** — audience validation on everywhere; SMART scope enforcement; close unauthenticated/spoofable write paths; identity from token not body; deployed-UI OIDC/PKCE login; TLS/mTLS; secrets manager (out of committed `.env`). *(closes §6 security gaps)*
- **Deploy pipeline** — build all images (incl. interop); helm/compose deploy + environment promotion. *(closes §6 no deploy pipeline)*
- **Observability** — OTel tracing + Prometheus metrics + central logs + DLQ replay tooling. *(closes §6 blind ops)*
- **Scale & tiers** — Temporal-worker population evaluation; Bulk Data `$export`; deployment-tier (pooled/siloed/boundary) made real beyond schema with per-boundary inference; X12 275 + CMS API entry (Plan-Net, PA API); performance NFRs; WCAG AA; cross-product audit/evidence-package export.

**Exit criteria:** GA release criteria met across products for the in-scope LOB/program; the stack deploys to staging by a non-developer; a request traces across services.

> **Security fork reminder:** the edge-auth + secrets + deploy items above don't depend on value-chain depth. If a real pilot must run on shared infrastructure (real PHI, external access), pull this subset forward to **immediately after Phase 0**. Under the chosen "design-partner pilot" north star they remain in Phase 6 (pilot assumed in a controlled/single-tenant boundary).
