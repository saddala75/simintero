Simintero Payer Platform

A unified prior-authorization, clinical-rules, AI-advisory, and quality-measure platform for payers. Polyglot monorepo, fully containerized, 46 services in a single docker compose up.

🐍 Python / FastAPI ☕ Java / Spring Boot ⬡ Node.js / TypeScript ⚛ React / Vite 🔧 Kafka · Postgres · Temporal · Keycloak · OPA · MinIO

Four Products, One Platform

:8000 Python

Enstellar — Prior Authorization & UM

End-to-end PA lifecycle: intake → auto-determination → clinical review → adverse decision → notification. Kafka-driven state machine with SLA clocks, RFI, escalation, appeals, and grievances.

:8083 JavaNode

Digicore — Clinical Rules Engine

CQL (Clinical Quality Language) rule authoring, versioning, governance approval, and real-time evaluation via ELM interpreter. Integrates with VKAS for versioned artifact storage.

:3014 Node

Revital AI — Evidence Pipeline

Temporal-orchestrated multi-step AI advisory: fetches clinical docs → extracts entities → maps to evidence criteria → generates grounded summaries → persists advisory. PHI-isolated at model boundary.

:3015 Node

Qualitron — Quality Measures

HEDIS-style quality measure evaluation, denominator/numerator/exclusion gap detection, outreach task creation, and aggregated reporting. Driven by Kafka lifecycle events.

Quick Start

Everything runs locally via Docker Compose. You need Docker Desktop with ≥8 GB RAM allocated.

First Boot

# Clone and start — takes 5–15 min on first run (HAPI loads IGs)
git clone https://github.com/saddala75/simintero.git
cd simintero
make up-build   # builds images + starts all 46 services

# Run the full E2E smoke test (~5 min)
make smoke

After first boot, images are cached and volumes persist, so subsequent starts take ~60–90 seconds:

make up      # start (fast)
make down    # stop, keep data volumes
make clean   # stop + wipe all volumes (clean slate)

All connections go through PgBouncer (transaction-mode pooler) on port 5433, which maps to Postgres on 5432. This allows dozens of services to share a small pool of actual Postgres connections.

Row-Level Security (RLS)

Every tenant's data is isolated at the database level using Postgres RLS. Every table in the workflow database has a tenant_id column and an RLS policy like:

CREATE POLICY tenant_isolation ON workflow_instances
  USING (tenant_id = current_setting('app.tenant_id'));

-- Every query sets this before executing:
SET LOCAL app.tenant_id = 'tenant-dev';

⚡ Invariant: BYPASSRLS

The sim_relay role has BYPASSRLS and is used only by the OutboxRelay and SlaPoller for cross-tenant reads. Application code uses workflow role (RLS enforced). To use sim_relay, you must explicitly SET ROLE sim_relay — workflow is a member but does NOT inherit BYPASSRLS automatically.

Transactional Outbox Pattern

This is the most important reliability pattern in the platform. No service writes to Kafka directly. Instead, every event is written to the shared.outbox table in the same database transaction as the business data change. A background poller then relays it to Kafka.

Why This Matters

🚨

Without the outbox: If a service writes to Postgres then crashes before publishing to Kafka, the event is lost and the system diverges silently. With the outbox, the event is durably recorded and will be published when the relay next runs — even if the service was down for hours.

How It Works

# 1. Business code writes event to outbox IN THE SAME TRANSACTION:
async with tenant_transaction(conn, tenant_id) as txn:
    await conn.execute("UPDATE workflow_instances SET status=$1 WHERE id=$2", state, case_id)
    await outbox.append_event(conn, CaseStateChanged(case_id=..., to_state=state))
    # ↑ Both succeed or both roll back together

# 2. OutboxRelay polls shared.outbox every 1 second:
SELECT * FROM shared.outbox
  WHERE published_at IS NULL AND dlq_at IS NULL
  FOR UPDATE SKIP LOCKED   -- safe for multiple relay instances
  LIMIT 100;

# 3. Relay publishes to Kafka, then marks published:
await producer.send(topic=row["topic"], value=row["envelope"])
UPDATE shared.outbox SET published_at = now() WHERE id = row["id"]

Relay Failure Handling

ScenarioBehaviour
Kafka publish failsRetried; after MAX_RELAY_RETRIES=5 → sets dlq_at on the row
Consumer processes event twiceDeduplicated via shared.processed_events — idempotent consumers
Consumer fails permanentlyAfter 3 retries → message to shared.consumer_dlq + Kafka dead-letter topic

Enstellar — Prior Authorization & UM Engine

The workflow engine is a Python/FastAPI application (services/enstellar-workflow/) that runs the full PA lifecycle. It owns the PA state machine, SLA clocks, RFI, escalation, appeals, and grievances.

Application Startup

The FastAPI lifespan context manager in main.py wires everything at startup:

@asynccontextmanager
async def lifespan(app):
    # Two asyncpg pools:
    workflow_pool = await create_pool(WORKFLOW_DB_URL)     # PA cases, RLS
    fabric_pool  = await create_pool(SIMINTERO_DB_URL)     # shared simintero DB

    # 9 background asyncio tasks (consumers + outbox relay):
    tasks = [
        IntakeConsumer(pool=workflow_pool).start(),
        ClinicalReviewConsumer(pool=workflow_pool).start(),
        AutoDeterminationConsumer(pool=workflow_pool).start(),
        RfiResponseConsumer(pool=workflow_pool).start(),
        QualGapClosedConsumer(pool=fabric_pool).start(),
        DecisionRecordedConsumer(pool=workflow_pool).start(),
        NotificationDeliveryConsumer(pool=workflow_pool).start(),
        OutboxRelay(pool=workflow_pool, producer=producer).start(),
        KafkaProducer().start(),
    ]
    yield
    # cancel all tasks on shutdown

PA State Machine

States

intake_received auto_determination clinical_review pend_rfi escalated approved denied partially_denied adverse_modification withdrawn closed

Transition Engine

engine/transitions.py — TransitionEngine.apply() runs these checks in order on every state change:

def apply(conn, req: TransitionRequest):
    case = await fetch_case(conn, req.case_id, req.tenant_id)

    # Guard 1: no transitions out of terminal states
    if case.status == "closed":
        raise TerminalStateError

    # Guard 2: adverse states require human signoff recorded first
    adverse_transition_guard(req.to_state, req.human_signoff_recorded)

    # Guard 3: OPA adverse-action policy (for denied/partially_denied/adverse_modification)
    if req.to_state in ADVERSE_STATES:
        opa_gate("sim/guards/adverse_action/allow", req)

    # Write immutable audit row, update status, publish outbox event
    await conn.execute("INSERT INTO workflow_events …")
    await conn.execute("UPDATE workflow_instances SET status=$1 …", req.to_state)
    await outbox.append_event(conn, CaseStateChanged(…))

⚡ Invariant #1 — Adverse decisions require human signoff

ADVERSE_STATES = {"denied", "partially_denied", "adverse_modification"}
These states also require OPA approval. AutoDeterminator can only produce approved or clinical_review — it can never auto-deny.

⚡ Invariant #2 — Terminal states are final

Once a case reaches closed, no further transitions are allowed. Other terminal states (approved, denied, etc.) can be re-examined via appeals, which create a separate appeal record — they do not reopen the original case.

SLA Clocks

Each case has a decision clock that tracks CMS regulatory deadlines. The clock pauses when an RFI is dispatched (provider has the ball) and resumes when the response arrives.

# In CaseService.pend_rfi() — all three happen in ONE transaction:
async with tenant_transaction(conn, tenant_id):
    transition(conn, case_id, to_state="pend_rfi")   # state change
    pause_clock(conn, case_id)                       # stop the timer
    dispatch_rfi(conn, rfi_request)                  # emit RFIDispatched event

The SlaPoller runs every 5 seconds (dev) / 30 seconds (prod). It uses SET ROLE sim_relay (BYPASSRLS) to scan all tenants' clocks and calls escalate(breach_mode=True) on any case whose deadline has passed.

Kafka Consumers

All consumers extend IdempotentKafkaConsumer, which provides deduplication, retry logic, and dead-lettering. They run as async tasks inside the workflow engine process.

IdempotentKafkaConsumer Base

class IdempotentKafkaConsumer:
    # Dedup: insert event_id into shared.processed_events before handling
    # If already present → skip (at-most-once per event)
    max_retries = 3

    async def _mark_processed(conn, event_id):
        try:
            await conn.execute("INSERT INTO shared.processed_events(event_id) VALUES($1)", event_id)
        except UniqueViolation:
            return False  # already processed
        return True

    # After max_retries failures → DLQ + Kafka dead-letter topic
    # enable_auto_commit=False — manual commit after _mark_processed

⚡ Invariant — ClinicalReview NEVER BLOCKS

If Revital AI submission fails, ClinicalReviewConsumer catches the error, emits AGENT_ASSIST_FAILED to the outbox, and returns success to the Kafka consumer. The case proceeds to clinical review without AI advisory. Reviewers are shown a "AI unavailable" indicator in the UI.

⚡ PHI Boundary — What Crosses the Revital Interface

Only procedure_codes, diagnosis_codes, urgency, and lob are sent to Revital. Member PHI (name, DOB, member ID) never crosses this boundary. Revital fetches docs via document_refs (opaque IDs), not by member identity.

FHIR Interop Layer

A Java/Spring Boot service (services/enstellar-interop/) that speaks the DaVinci Prior Authorization Support (PAS) FHIR IG to EHRs, and translates to the platform's internal event model.

PAS $submit Flow

When an EHR submits a PA request via POST /fhir/Claim/$submit:

  1. 1

    Store-First

    Raw FHIR bundle immediately stored to MinIO (enstellar-raw-bundles). If anything fails after this, the raw bundle can be replayed.

  2. 2

    Validate

    Bundle validated against DaVinci-PAS FHIR profile using HAPI validator. Reject 422 if invalid.

  3. 3

    Normalize

    POST http://workflow-engine:8000/normalize — extracts procedure codes, diagnosis codes, member ref, urgency, LOB from the bundle into the platform's canonical model.

  4. 4

    Idempotency Check

    DecisionStore.save() — idempotent on correlationId. Returns 422 on duplicate submission (same correlationId resubmitted).

  5. 5

    Publish to Kafka

    CaseIntakeReceived published to sim.case.lifecycle. This is what the IntakeConsumer picks up.

  6. 6

    Return QUEUED

    Returns a FHIR ClaimResponse bundle with outcome=QUEUED. EHR polls status via GET /fhir/Claim/{correlationId}/$inquire.

Revital AI — Evidence Pipeline

Revital takes clinical documents for a case and produces an AI-generated advisory: a grounded summary of whether the submitted clinical evidence supports the requested procedure. It runs as a Temporal workflow so partial failures are retried without starting over.

Model Gateway — AI Inference Proxy

Revital never calls Anthropic directly. It goes through the model-gateway (:3011), which:

  • Checks the OPA kill-switch (operator can disable AI without a deploy)
  • Runs the PHI filter (allow-list scrubbing before anything reaches Anthropic)
  • Resolves the model binding from VKAS (which Claude model version, which prompt version)
  • Tracks FinOps metrics (token usage per tenant per model)

Qualitron — Quality Measures

Qualitron evaluates HEDIS-style quality measures, detects care gaps, creates outreach tasks, and provides aggregated reporting. It's driven primarily by Kafka events from the case lifecycle.

Gap Detection Logic

From modules/qualitron/gaps/src/GapDetector.ts:

function detectGap(input: GapInput): GapDecision {
  // Not in denominator or explicitly excluded → no gap
  if (!input.denominator || input.exclusion) return { has_gap: false };

  // Numerator already hit → no gap (measure met)
  if (input.numerator) return { has_gap: false };

  // In denominator, not excluded, numerator not met → gap
  return {
    has_gap: true,
    gap_type: "missing_numerator",
    should_create_outreach: true
  };
}

End-to-End Walkthrough

Trace a PA request from EHR submission to member letter. Every service involved, every event fired.

Scenario: Shoulder arthroscopy (CPT 29826) — Denied

  1. 1

    EHR submits PA bundle

    POST http://interop:8080/fhir/Claim/$submit with DaVinci-PAS FHIR bundle. Interop stores raw bundle to MinIO, validates, normalizes via workflow-engine, then publishes CaseIntakeReceived to sim.case.lifecycle.

  2. 2

    IntakeConsumer creates the case

    IntakeConsumer picks up CaseIntakeReceived. Calls CaseService.create_case() — inserts row in workflow_instances (status=intake_received), starts decision clock. Publishes CaseStateChanged.

  3. 3

    Auto-determination

    AutoDeterminationConsumer runs AutoDeterminator: calls Digicore runtime (POST :8083/v1/runtime/coverage-discovery). CPT 29826 requires PA → transitions to clinical_review. Publishes CaseStateChanged to_state=clinical_review.

  4. 4

    ClinicalReviewConsumer triggers Revital

    Fetches document refs from DocumentService. Sends AnalysisInput to POST :3014/analyses (only codes + doc refs — no PHI). Revital starts Temporal workflow: fetch docs → extract entities → map to criteria → LLM grounded summary. Writes advisory to DB.

  5. 5

    Reviewer opens case in portal

    Reviewer opens /cases/{caseId}. BFF calls GET /workbench — parallel fetches: case details, Digicore criteria, Revital AI suggestions, FHIR docs. AI advisory shows "evidence insufficient: conservative therapy not documented."

  6. 6

    Medical director records signoff + denies

    Medical director clicks "Deny." BFF's POST /adverse-decision: validates clinician_id == JWT.sub, checks sign_off_confirmed=true. Calls record_signoff(), then calls OPA adverse-action gate (sim/guards/adverse_action/allow). Both pass → transitions to denied. Publishes CaseStateChanged + AdverseDetermination + DecisionRecorded.

  7. 7

    DecisionRecordedConsumer builds notice

    DecisionRecordedConsumer picks up DecisionRecorded. NotificationService renders a CMS-compliant denial letter with member PHI, stores it to MinIO (simintero-letters). Publishes NotificationSent on sim.artifact with only a body_ref (MinIO key) — no PHI on the event bus.

  8. 8

    NotificationDeliveryConsumer sends email

    Picks up NotificationSent. Routes to SmtpEmailSender → Mailpit (:1025). In dev, the email is visible at http://localhost:8025. Updates notification_log.delivered_at.

  9. 9

    EHR polls for decision

    GET /fhir/Claim/{correlationId}/$inquire → returns FHIR ClaimResponse with outcome=DENIED. Case is terminal — clock stopped at step 6 when denied.
