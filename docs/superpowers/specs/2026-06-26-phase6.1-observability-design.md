# Phase 6.1 — Observability Design

**Date:** 2026-06-26
**Status:** Approved

## Goal

Add end-to-end observability (traces, metrics, logs) across all 24+ Simintero services using OpenTelemetry auto-instrumentation, a self-hosted backend stack (Prometheus + Tempo + Loki + Grafana), and PHI scrubbing at the collector layer. Runs in both docker-compose (local dev) and Kubernetes (production).

## Architecture

```
Services (Python / Node.js / Java)
  │  auto-instrumentation (HTTP, DB, Kafka spans)
  │  + enrichment middleware (tenant_id, case_id on active span)
  ▼
OTel Collector  (OTLP gRPC :4317, HTTP :4318)
  │  transform/scrub_phi processor  ← PHI stripped here, before any export
  │  batch processor
  ├──► Prometheus remote-write  →  Prometheus
  ├──► OTLP/gRPC               →  Tempo
  └──► Loki push               →  Loki
                                       ▼
                                 Grafana  (Prometheus + Tempo + Loki datasources)
```

Both environments share the same OTel Collector config file. Deployment differences:

- **docker-compose** — five new services (`otel-collector`, `prometheus`, `tempo`, `loki`, `grafana`) added to `docker-compose.yml`. All existing services get `OTEL_EXPORTER_OTLP_ENDPOINT: http://otel-collector:4317` and `OTEL_SERVICE_NAME: <name>`.
- **Kubernetes** — `observability` namespace; five Helm charts (community charts for each backend + custom chart for collector config). Services reference `otel-collector.observability.svc:4317`. Grafana datasources and dashboards provisioned as ConfigMaps.

## Language Instrumentation

### Python (`workflow-engine`, `portal-bff`)

**Dependencies added to `pyproject.toml`:**
```
opentelemetry-sdk
opentelemetry-exporter-otlp-proto-grpc
opentelemetry-instrumentation-fastapi
opentelemetry-instrumentation-asyncpg
opentelemetry-instrumentation-httpx
opentelemetry-instrumentation-logging
```

**Bootstrap in `main.py`** before the FastAPI app is created:
```python
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.instrumentation.fastapi import FastAPIInstrumentor
from opentelemetry.instrumentation.asyncpg import AsyncPGInstrumentor
from opentelemetry.instrumentation.httpx import HTTPXClientInstrumentor

provider = TracerProvider()
provider.add_span_processor(BatchSpanProcessor(OTLPSpanExporter()))
trace.set_tracer_provider(provider)
FastAPIInstrumentor().instrument()
AsyncPGInstrumentor().instrument()
HTTPXClientInstrumentor().instrument()
```

**Enrichment middleware** — a FastAPI middleware reads `tenant_id` and `sub` from the request's already-validated auth context (populated by `require_auth`) and stamps them onto the active span. Zero per-route changes required.

```python
@app.middleware("http")
async def otel_enrich(request: Request, call_next):
    span = trace.get_current_span()
    if ctx := getattr(request.state, "tenant_context", None):
        span.set_attribute("tenant_id", ctx.tenant_id)
        span.set_attribute("user.sub", ctx.sub)
    return await call_next(request)
```

### Node.js (15+ services)

**New shared workspace package `@sim/otel`** (`platform/packages/otel/`):
```typescript
// src/index.ts
import { NodeSDK } from '@opentelemetry/sdk-node'
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node'
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-grpc'
import { OTLPMetricExporter } from '@opentelemetry/exporter-metrics-otlp-grpc'
import { PeriodicExportingMetricReader } from '@opentelemetry/sdk-metrics'

const sdk = new NodeSDK({
  traceExporter: new OTLPTraceExporter(),
  metricReader: new PeriodicExportingMetricReader({
    exporter: new OTLPMetricExporter(),
    exportIntervalMillis: 15_000,
  }),
  instrumentations: [getNodeAutoInstrumentations()],
})
sdk.start()
```

Each Node.js service adds one line to its entrypoint (before all other imports):
```typescript
import '@sim/otel'
```

**Enrichment middleware** added to the `base-service` Express pattern:
```typescript
app.use((req, res, next) => {
  const span = trace.getActiveSpan()
  if (span && req.tenantId) {
    span.setAttribute('tenant_id', req.tenantId)
    span.setAttribute('user.sub', req.sub ?? '')
  }
  next()
})
```

`getNodeAutoInstrumentations()` covers: Express/Fastify (HTTP spans), `pg` (DB queries), `kafkajs` (producer/consumer spans), `axios`/`undici` (outbound HTTP), DNS.

### Java (`digicore-runtime`, `interop`)

**Java agent approach** — download `opentelemetry-javaagent.jar` in the Dockerfile and pass `-javaagent` to the JVM. No code changes required for auto-instrumentation.

```dockerfile
ADD https://github.com/open-telemetry/opentelemetry-java-instrumentation/releases/download/v2.6.0/opentelemetry-javaagent.jar /app/opentelemetry-javaagent.jar
ENV JAVA_TOOL_OPTIONS="-javaagent:/app/opentelemetry-javaagent.jar"
```

The agent auto-instruments: Spring MVC (HTTP spans), JDBC (DB query spans), Kafka producer/consumer, Spring RestTemplate/WebClient (outbound HTTP).

**Tenant enrichment** via a Spring `HandlerInterceptor`:
```java
@Component
public class OtelTenantInterceptor implements HandlerInterceptor {
    @Override
    public boolean preHandle(HttpServletRequest req, HttpServletResponse res, Object handler) {
        String tenantId = (String) req.getAttribute("tenantId"); // set by JWT filter
        if (tenantId != null) {
            Span.current().setAttribute("tenant_id", tenantId);
        }
        return true;
    }
}
```

## OTel Collector Configuration

Single config file (`infra/otel/collector-config.yaml`) shared between compose and K8s:

```yaml
receivers:
  otlp:
    protocols:
      grpc:   # :4317
      http:   # :4318

processors:
  batch:
    timeout: 5s
    send_batch_size: 1024

  transform/scrub_phi:
    trace_statements:
      - context: span
        statements:
          - delete_key(attributes, "member_ref")
          - delete_key(attributes, "resolution")
          - delete_key(attributes, "description")
          - delete_key(attributes, "filing_body")
          - delete_key(attributes, "body_ref")
          - delete_key(attributes, "member_phi")
    log_statements:
      - context: log
        statements:
          - delete_key(attributes, "member_ref")
          - delete_key(attributes, "resolution")
          - delete_key(attributes, "description")

exporters:
  otlp/tempo:
    endpoint: tempo:4317
    tls:
      insecure: true
  prometheusremotewrite:
    endpoint: http://prometheus:9090/api/v1/write
  loki:
    endpoint: http://loki:3100/loki/api/v1/push
  logging:         # dev only — shows spans in collector stdout
    verbosity: detailed

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [transform/scrub_phi, batch]
      exporters: [otlp/tempo]
    metrics:
      receivers: [otlp]
      processors: [batch]
      exporters: [prometheusremotewrite]
    logs:
      receivers: [otlp]
      processors: [transform/scrub_phi, batch]
      exporters: [loki]
```

The `transform/scrub_phi` processor runs **before** `batch` and **before** any exporter. PHI attribute names are deleted in-process; no PHI ever reaches Tempo, Prometheus, or Loki. Adding a new PHI field to scrub requires only a one-line collector config change — no service restarts.

K8s override: the `exporters` section uses cluster-internal DNS (`tempo.observability.svc`, `prometheus.observability.svc`, `loki.observability.svc`) instead of bare hostnames.

## Grafana Dashboards

Four dashboards committed as JSON provisioning files under `infra/grafana/dashboards/`. Grafana auto-loads them at startup via a ConfigMap volume mount — no manual import.

### 1. PA Lifecycle SLA
- Case intake → determination latency (p50/p95/p99 by tenant)
- Cases approaching SLA breach (count by tenant/LOB)
- Determination outcome breakdown (auto-approve / clinical-review / adverse)
- RFI round-trip time (filed → response)

### 2. Service Health
- Per-service error rate (5xx and 4xx), request throughput, p95 latency
- One row per service; data sourced from HTTP span metrics auto-captured by instrumentation
- Prometheus → Tempo exemplar links for jumping from a slow-request panel directly to its trace

### 3. Infrastructure
- Kafka consumer lag per topic (cases, grievances, appeals, evidence, sim.evidence)
- DB query p95 latency by service (from asyncpg / pg / JDBC auto-instrumentation)
- Temporal worker task-queue backlog

### 4. Trace Explorer
- Grafana Explore view pre-configured with the Tempo datasource
- Default filter: `service.name =~ "simintero.*"`
- Linked from Service Health panels via Tempo exemplars

## PHI Scrubbing — Field Inventory

Fields deleted from span attributes and log attributes before export:

| Field | Source service |
|---|---|
| `member_ref` | workflow-engine, portal-bff, digicore-runtime |
| `resolution` | workflow-engine (grievances) |
| `description` | workflow-engine (grievances, cases) |
| `filing_body` | workflow-engine |
| `body_ref` | workflow-engine (PHI letter MinIO key) |
| `member_phi` | workflow-engine |

New PHI fields discovered during development are added to `infra/otel/collector-config.yaml` — one-line change, no service deploys required.

## Validation

**PHI scrubber integration test** — a synthetic span with `member_ref: "test-member-001"` is sent to a test collector instance running the production config; the exported span is asserted to have no `member_ref` attribute. Fails if the scrubber is misconfigured.

**OTel package smoke test** — the `@sim/otel` Node.js package sends a synthetic span to a local OTLP receiver and asserts it is received. Python equivalent uses `opentelemetry-sdk`'s in-memory exporter.

**Dev logging exporter** — the collector's `logging` exporter (enabled in compose, disabled in K8s) outputs received spans to `docker compose logs otel-collector`, giving developers immediate visibility during local development without needing to open Grafana.
